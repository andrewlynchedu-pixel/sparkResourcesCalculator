# Описание Spark Capacity Calculator

Этот калькулятор — профессиональный инструмент для Data Engineer-а, основанный на концепции «вычислительных волн» (execution waves) и модели Unified Memory Management, появившейся в Spark 1.6.

Он переводит абстрактные бизнес-требования (объем данных и желаемое время) в физические ресурсы YARN или Kubernetes (ядра, гигабайты Heap и Off-Heap памяти).

Ниже приведено подробное описание каждого параметра калькулятора: на что он влияет, что означает физически и как получить его значение в реальной жизни.

---

## Группа 1: Бизнес-требования (вводные от клиента)
Эти параметры меняются для каждого нового запроса на создание витрины.

### 1) `data_size_gb` (объем входных данных)
- **Что значит:** физический размер данных на диске (в HDFS/S3), которые нужно прочитать для расчета витрины.
- **На что влияет:** вместе с размером блока (партиции) определяет общее количество элементарных задач (`Tasks`).
- **Как получить:** посмотреть размер директории с исходными parquet-файлами.

### 2) `sla_seconds` (Service Level Agreement)
- **Что значит:** желаемое или жестко заданное бизнесом время расчета витрины в секундах.
- **На что влияет:** определяет, на сколько «итераций» (волн) кластер может растянуть выполнение задач. Чем меньше SLA, тем меньше доступных волн, и тем больше экзекьюторов нужно запустить параллельно.

---

## Группа 2: Системные константы (настройки архитектуры)
Задаются один раз при инициализации калькулятора под ваш конкретный кластер.

### 3) `block_size_mb` (размер блока HDFS / партиции)
- **Что значит:** объем данных, который обрабатывается одной задачей (`Task`) на одном ядре.
- **На что влияет:** формирует количество `Tasks`: `N_tasks = Data / BlockSize`.
- **Как получить:** обычно это стандартный размер блока в файловой системе (по умолчанию 128 МБ). Менять стоит редко.

### 4) `cores_per_exec` (количество ядер на экзекьютор)
- **Что значит:** сколько задач (потоков) один JVM-контейнер выполняет параллельно.
- **На что влияет:** на плотность упаковки задач. Рекомендуется 4–5 ядер. 1–2 ядра дают избыточные накладные расходы JVM, более 5 ядер часто приводят к конкуренции за сеть и I/O bottleneck.
- **Как получить:** зафиксировать как стандарт (например, 5 vCPU).

### 5) `driver_mem_gb` и `driver_cores` (ресурсы драйвера)
- **Что значит:** ресурсы узла, который строит DAG и раздает задачи.
- **На что влияет:** гарантирует, что планировщик YARN выделит драйверу отдельный контейнер.
- **Как получить:** обычно хватает 2–4 ГБ и 2–4 ядер. Если часто используются `collect()` или `BroadcastHashJoin`, память драйвера увеличивают до 8–16 ГБ, иначе возможен `OutOfMemoryError`.

## Группа 3: Эмпирические параметры (телеметрия Spark)
Отражают поведение инфраструктуры и форматов данных.

### 6) `t_base_sec` (базовое время задачи)
- **Что значит:** за сколько секунд процессор в среднем читает и выполняет простой `SELECT *` для одного блока данных (например, 128 МБ) без сложных преобразований.
- **На что влияет:** это фундамент для расчета длительности одной «волны».
- **Как получить:** выполнить простой Spark-запрос (чтение + запись) на 10–20 ГБ, затем в Spark UI (`Stages` → `Summary Metrics`) посмотреть `Duration` в колонке `Median`.

### 7) `e_mem_rate` (коэффициент расширения памяти)
- **Что значит:** во сколько раз данные «распухают» при чтении в память относительно размера на диске.
- **На что влияет:** напрямую определяет, сколько RAM нужно запросить для одного экзекьютора.
- **Как получить:** в Spark UI для тяжелого stage разделить `Shuffle spill (memory)` на `Shuffle spill (disk)`. Обычно диапазон 2.0–5.0.

---

## Группа 4: Профилирующие коэффициенты (специфика витрины)
Позволяют калькулятору учитывать сложность SQL-кода и распределения данных.

### 8) `k_comp` (коэффициент вычислительной сложности / Compute)
- **Что значит:** показывает, насколько алгоритм сложнее простого чтения.
- **На что влияет:** увеличивает базовое время (`T_base × K_comp`) и требуемый объем RAM на задачу.
- **Как определить:**
  - Экспертно: `1.0` (простая фильтрация), `1.5` (агрегации), `2.0–3.0` (тяжелые window-функции и `SortMergeJoin`).
  - Из Spark UI: `Median Task Duration` на самом тяжелом этапе / `t_base_sec`.

### 9) `k_skew` (коэффициент перекоса данных / Skew)
- **Что значит:** степень неравномерности распределения данных по ключам партиционирования.
- **На что влияет:** увеличивает эффективное время волны. При сильном skew (`> 2.0`) линейное добавление ядер часто не спасает SLA из-за `straggler tasks`.
- **Как определить:** в Spark UI (`Summary Metrics`) взять `Max Task Duration / Median Task Duration`.
- **Практика:** при высоком skew включать `spark.sql.adaptive.skewJoin.enabled` (AQE).

---

## Группа 5: Внутренние константы памяти Spark
Эти параметры зашиты в архитектуру Spark 1.6+ (Unified Memory Manager) и YARN/K8s.

- `reserved_memory_mb = 300`: жестко зарезервированные 300 МБ Heap для внутренних нужд Spark.
- `memory_fraction = 0.6`: 60% Heap выделяется под Unified Memory (execution + storage), остальные 40% — user memory и метаданные.
- `overhead_fraction = 0.10`: помимо Heap контейнеру нужна Off-Heap память (OS, Netty, PySpark), по умолчанию 10% от Heap, но не менее 384 МБ.

---

## Пример полного расчета (как это работает внутри)
**Дано:** 100 ГБ данных, SLA 10 минут (600 сек), `k_comp = 1.5`, `k_skew = 1.2`, блок 128 МБ, `t_base_sec = 25`, `e_mem_rate = 3`, `cores_per_exec = 5`.

1. **Количество задач:** `100 × 1024 / 128 = 800 tasks`.
2. **Время одной задачи/волны:** `25 × 1.5 × 1.2 = 45 сек`.
3. **Доступные волны в SLA:** `600 / 45 = 13.3`, округляем вниз → `13`.
4. **Нужные ядра:** `800 / 13 = 62` параллельных ядра.
5. **Нужные экзекьюторы:** `62 / 5 = 12.4`, округляем вверх → `13 executors`.
6. **Память на одну задачу:** `128 × 3 × 1.5 = 576 МБ`.
7. **Unified memory на экзекьютор:** `576 × 5 = 2880 МБ`.
8. **Heap:** `(2880 / 0.6) + 300 = 5100 МБ` → `spark.executor.memory`.
9. **Overhead:** `10% × 5100 = 510 МБ` (больше 384) → `spark.executor.memoryOverhead`.
10. **Контейнер YARN/K8s:** `5100 + 510 = 5610 МБ (~5.5 ГБ)`.

Итог: вы получаете обоснованный `spark-submit` конфиг, который помогает уложиться в SLA и снизить риск `OutOfMemoryError`.
